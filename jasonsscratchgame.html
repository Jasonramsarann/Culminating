Jason's Scratch Game W/ Progress



Introduction

For my Scratch game, I decided to create a car parking simulator. I got the idea a few days ago while driving with a driving instructor. Since I am practicing for my upcoming G2 license test, I figured that a driving and parking simulator would be helpful. I mainly struggle with my parking, especially my reverse parking. This game serves a purpose because it can teach people what kind of angles they need to approach a parking space with to have a good parking. It also builds good habits by teaching people that it is important to stay within the lines of a parking space. The neat thing is that you are able to do front, reverse and sometimes parallel parking while using my program.

Progress Photo Progress Day:
On the first day, I created a Scratch account. I started the project off by brainstorming an idea for my assignment. In preparation for the assignment, I read online forums about Scratch and how the blocks function. After brainstorming, I came up with the idea of a parking simulator. I created two sprites: one for the directional arrow and one for the car. I input basic code that would allow the car to follow the directional arrow. To move the car, you would have to use the cursor for direction, the up arrow key to move forward and the down arrow key to move backwards. I also created variables that would allow the car to accelerate. 1
On day two, I created variables that would allow all of the keyboard arrows to function. I added blocks that would allow me to imitate an x and y joystick which would allow me to use all four arrow keys to move. I also created three new variables which would allow me to brake more smoothly, speed up and move more powerfully. I noticed that the directional arrow would lag whenever it was given a command to move. To fix this, I used a broadcast command which would allow the car body sprite to activate first. This removed the directional arrow lag completely. 2
On day three, I started creating the obstacle course and the framework for the game. The idea behind the game is that you must park within the parking space without hitting any obstacles. For the obstacle sprite, I created code that would cause the level to reset each time that the car hit an obstacle. For the parking space sprite, I created two costumes; one that would serve as the visual parking space and invisible rectangle within the parking space. The idea behind this is that whenever the car is within the invisible rectangle, it would cause the player to win. The purpose of this was to ensure that a player could not win if they were touching any of the lines of the parking space. I also created a sprite that would have a "You Win" message each time the player was inside the invisible rectangle. 3
On the final day, I added details like additional levels, colours, functions to make driving smoother and small details like a horn and radio function. I created four levels for my game, each with increasing difficulty. I had to undergo a rigorous testing process to make sure that users are able to complete each level. I also had to test my functions to make sure that they worked and that there were no glitches. For some detail work, I added in a functional horn using a new sprite, some code and a generic horn sound from the Scratch Program. I also added a 'radio' which plays a beat that I made. Users are able to customize the volume and pitch of the beat as well. Overall, it a very busy day. 4

Testing Process

I went through a very lengthy testing process to figure out if there were any glitches and find out how I could correct them. Of course, there were glitches in my program that I had to work around. After every block of code that I placed, I tried to test it to see if it was working properly. For the most part, every part worked smoothly.

Problems During The Testing Process

Issue Resolution
Level change not working I originally had a system in place that would let the user automatically move onto the next level after they passed a level. For some reason, the system decided to stop working and I was unable to diagnose the exact source of the problem. Because of this problem, the user would only be able to progress to the next level half of the time. Sometimes users would get stuck on the first level and a have no way to progress. I worked around this issue by putting two systems in place. Firstly, I put a green button at the end of each course. When the user completes the course, they have to hit the green button with their car to move onto the next level. The second measure that I put into place was a manual override. If the user presses the start button and presses the letter c, then this will allow them to skip or change levels.
Directional Arrow Lag On day two of my Scratch game, there was a good amount of lag between the movement of the directional arrow and the car. Whenever I would move the directional arrow, the car would lag one frame behind. To fix this, I used a broadcast block which would activate the car commands slightly before the arrow commands. This completely fixed the lag and helped them work together smoothly.
Parking Spot Not Working Initially, I only had the outline for the parking space. I noticed that by doing this, anybody could win by simply placing their car on the parking spot line. I addressed this issue by making a rectangle within the parking space. I had to make sure that the rectangle was smaller than the parking space, but bigger than the car. By doing this, the user could only pass levels by orienting their car within the rectangle. To make the parking space more visually appealing, I made the rectangle invisible.
